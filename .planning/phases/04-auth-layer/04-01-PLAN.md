Create `app/proxy.ts` (NOT middleware.ts — Next.js 16 renamed it per RESEARCH.md):

```typescript
import { createClient } from '@/lib/supabase/proxy'
import { NextResponse, type NextRequest } from 'next/server'

export async function proxy(request: NextRequest) {
  // Create response first, then modify
  let response = NextResponse.next({
    request: { headers: request.headers },
  })

  const { supabase, response: updatedResponse } = createClient(request, response)
  response = updatedResponse

  // CRITICAL: Use getUser(), not getSession() — validates JWT
  // This refreshes the session if needed
  await supabase.auth.getUser()

  return response
}

export const config = {
  matcher: [
    // Match all routes except static files and images
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
}
```

Key points per RESEARCH.md and D-01/D-02:
- Export `proxy` function (not `middleware`)
- Call `getUser()` to trigger session refresh
- No redirects — just refresh tokens and pass through
- Use correct matcher to skip static assets

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> proxy.ts | Untrusted cookies arrive from browser |
| proxy.ts -> Supabase | Token validation against Supabase Auth |
| Server Component -> Client Component | initialUser passed as prop |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | Session cookies | mitigate | Use getUser() not getSession() — validates JWT against Supabase |
| T-04-02 | Tampering | Cookie values | mitigate | Supabase JWT signature verification in getUser() |
| T-04-03 | Information Disclosure | Cross-subdomain cookies | accept | Cookie-Domain restricted to .generation-ai.org — trusted subdomains only |
| T-04-04 | Denial of Service | Token refresh failure | mitigate | Graceful degradation to V1 mode on auth errors |
</threat_model>
